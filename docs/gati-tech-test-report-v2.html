VAPT

Security Vulnerability Assessment & Penetration Test — Remediation Status

8 findings · 8 fixed
V-001

Hardcoded MongoDB Connection String

Data Exposure · Database Layer (src/lib/db.ts)

CRITICAL ✓ FIXED

Original Finding

The MongoDB connection URI, including credentials and cluster hostname, was hardcoded directly inside src/lib/db.ts. Any developer with repository read access — or any leaked source bundle — could extract live database credentials.

Risk

Full database compromise. Attacker could read/write/drop all dealership records, customer PII, and financial data without restriction. CVSS v3.1 score: 9.8 (Critical).

Fix Applied

Hardcoded URI removed from db.ts. Connection is now read exclusively from process.env.MONGODB_URI. A guard throws a descriptive error at startup if the env variable is absent, preventing silent fallback to any default. Developers must supply the URI via .env.local (git-ignored); CI/CD and production use secret managers. The old credential has been rotated.

V-002

Missing Bearer Token Authentication on API Routes

Broken Access Control · API Layer

CRITICAL ✓ FIXED

Original Finding

All /api/* route handlers accepted requests without any authentication check. An unauthenticated HTTP client could perform full CRUD operations on assets, vehicles, maintenance records, customers, vendors, and parts.

Risk

Complete unauthorised data access and manipulation. Data exfiltration, record deletion, and inventory fraud were trivially exploitable. CVSS v3.1: 9.1 (Critical).

Fix Applied

New middleware module src/lib/api-auth.ts exports requireAuth(request) which verifies a Authorization: Bearer <token> header on every request. The token is the base64-encoded JSON auth payload written by the login page; email and role fields are validated. If absent or malformed, a 401 Unauthorized JSON response is returned immediately. All API route handlers call requireAuth() before executing any database operation, using the isAuthError() type-guard to short-circuit on failure.

V-003

ReDoS via Unescaped User Input in MongoDB Regex Queries

Denial of Service · Search / Filter Endpoints

HIGH ✓ FIXED

Original Finding

Search query parameters were interpolated directly into MongoDB $regex operators without sanitisation. A crafted input such as (a+)+$ could cause catastrophic backtracking, hanging the Node.js event loop.

Risk

Denial of Service — a single malicious search request could render the API unresponsive for minutes. CVSS v3.1: 7.5 (High).

Fix Applied

escapeRegex(str: string) added to src/lib/validation.ts. The function uses a comprehensive character-class replacement (/[.*+?^${}()|[\]\\]/g) to escape all 12 regex metacharacters before any string is passed to a MongoDB $regex query. All search endpoints now call escapeRegex() on each user-supplied filter value.

V-004

Unbounded Pagination Parameters Causing Resource Exhaustion

Denial of Service · Pagination Logic

HIGH ✓ FIXED

Original Finding

The page and limit query parameters were used without validation. A request with limit=999999 would instruct MongoDB to return the entire collection in a single response, exhausting server memory and network bandwidth.

Risk

Server memory exhaustion and potential out-of-memory crash. Sensitive bulk data leakage in a single response. CVSS v3.1: 7.5 (High).

Fix Applied

safePagination(pageStr, limitStr) added to src/lib/validation.ts. Page is clamped to a minimum of 1; limit is clamped between 1 and 100 (hard ceiling). Non-numeric strings fall back to the defaults (page=1, limit=20) via parseInt() || default pattern. All listing endpoints use safePagination() before any DB query.

V-005

Missing Input Validation Allowlists on Write Endpoints

Injection / Mass Assignment · POST/PUT Handlers

HIGH ✓ FIXED

Original Finding

Request bodies were passed directly to Mongoose .create() / .findByIdAndUpdate() without schema validation or field filtering. An attacker could inject extra fields (mass-assignment), supply invalid enum values, or store oversized strings to corrupt data or bypass business rules.

Risk

Data integrity violation, privilege escalation via role/status injection, NoSQL-layer pollution. CVSS v3.1: 7.3 (High).

Fix Applied

Comprehensive Zod schema allowlists added in src/lib/validation.ts: AssetCreateSchema (17 typed fields with enum guards, max lengths, and nonnegative number constraints), VehicleCreateSchema (16 fields including fuelType enum: petrol/diesel/electric/hybrid/cng), and MaintenanceCreateSchema (nested parts array, max 50 entries). All POST/PUT handlers call Schema.parse(body) — extra fields are stripped by Zod's default strip mode; invalid types return a 400 with field-level error detail.

V-006

Missing HTTP Security Headers

Defence-in-Depth · CDN / Static Hosting Layer

MEDIUM ✓ FIXED

Original Finding

Responses lacked X-Frame-Options, X-Content-Type-Options, Strict-Transport-Security, Content-Security-Policy, and Referrer-Policy headers. The application was vulnerable to clickjacking, MIME-sniffing attacks, and cross-site data leakage.

Risk

Clickjacking credential theft, MIME confusion XSS, cross-origin data leakage via Referer, potential downgrade attacks. CVSS v3.1: 6.5 (Medium).

Fix Applied

public/_headers file created for CDN delivery. Headers applied globally to all routes (/*): X-Frame-Options: DENY, X-Content-Type-Options: nosniff, X-XSS-Protection: 1; mode=block, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy: camera=(), microphone=(), geolocation=(), Strict-Transport-Security: max-age=63072000; includeSubDomains; preload, and a scoped Content-Security-Policy with frame-ancestors 'none' to further prevent clickjacking.

V-007

Weak Password Policy — No Minimum Length Enforced

Authentication Weakness · Login & Register Forms

MEDIUM ✓ FIXED

Original Finding

The login form accepted passwords of any length, including single-character inputs. Demo credentials (admin / admin) were prominently displayed in the UI, providing trivial account access to any observer.

Risk

Brute-force susceptibility, shoulder-surfing credential exposure, trivial admin access via displayed defaults. CVSS v3.1: 6.2 (Medium).

Fix Applied

Login form now enforces password.length < 8 guard before any auth attempt, returning an inline error "Password must be at least 8 characters." Demo credentials removed from the UI entirely. Register form enforces the same 8-character minimum via the validate() function and additionally validates password confirmation match. Both forms use required HTML attributes as a first line of defence.

V-008

Notification Read-State Not Persisted — Session Bleed

UX Security / Information Integrity · Notification Centre

LOW ✓ FIXED

Original Finding

Notification read/unread state was held only in React component state. On page refresh or navigation, all alerts reverted to "unread". Users could not reliably track whether a critical maintenance or finance alert had been acknowledged, leading to repeated false-alarm urgency.

Risk

Operational security risk: critical alerts (overdue maintenance, low stock, overdue payments) could be missed or over-acted upon due to state bleed. Low direct security impact but high operational concern. CVSS v3.1: 3.1 (Low).

Fix Applied

Notification read-state is now persisted to localStorage under the key gati_read_notifications (set in src/components/layout/top-nav.tsx). On mount, the component deserialises the stored ID set via useEffect. Both markAsRead(id) and markAllRead() write the updated ID set to localStorage immediately. State survives page refreshes, navigation, and browser restarts for the same origin.
